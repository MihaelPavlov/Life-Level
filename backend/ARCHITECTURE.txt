MODULAR MONOLITH + PORTS & ADAPTERS — LIFE-LEVEL BACKEND
=========================================================

CONTEXT
-------
The current LifeLevel.Api project is a single-project ASP.NET Core app with 28 entities,
16 controllers, 13 services, and a 41-DbSet AppDbContext. As Phase 3-9 features land
(Guild Raids, Items, Achievements, Seasonal Events, Social), the project will become
difficult to navigate, test, and reason about.

Refactoring to a Modular Monolith gives each business domain its own project with explicit
contracts, while keeping a single deployable binary and a single database. Each module uses
Ports & Adapters internally — pure domain, interface-driven application layer, EF
implementations as adapters.

Single deployable: all modules compile into one LifeLevel.Api.exe. The modules are class
libraries with no Program.cs or HTTP listener. If a module needs to become a microservice
later, the boundaries are already clean.


CONCEPT GLOSSARY
----------------
WorldZone  — The overworld layer. Named regions (Forest of Endurance, Mountains of Strength,
             Ocean of Balance) connected by a zone graph (World → WorldZone → WorldZoneEdge).
             Users travel between zones; zone unlocks gate progression by level requirement.

Map        — The dungeon instance layer within a zone. A node/edge graph (MapNode → MapEdge)
             containing encounter nodes: Boss, Chest, DungeonPortal, and Crossroads.
             Users navigate the map by spending movement earned from real-world activity.

The two layers are separate modules. WorldZone owns overworld state (UserWorldProgress,
UserZoneUnlock). Map owns dungeon state (UserMapProgress, UserNodeUnlock). A user arrives
at a zone via WorldZone, then explores its internal map via Map + Adventure modules.

CROSS-MODULE COMMUNICATION: TWO-TIER PATTERN
---------------------------------------------

TIER 1 — Direct interface calls (synchronous, same transaction)
  Used when: caller needs the result in the same HTTP response.

  All modules that award XP     -> ICharacterXpPort.AwardXpAsync()
  LoginRewardService            -> IStreakShieldPort.AddShieldAsync()
  CharacterService.GetProfile   -> IStreakReadPort, ILoginRewardReadPort, IDailyQuestReadPort

TIER 2 — In-process domain events (fire-and-await, after save)
  Used when: side effect does not affect the response and direct coupling would create a cycle.

  ActivityService publishes ActivityLoggedEvent
    -> StreakActivityHandler  (Streak module)
    -> QuestActivityHandler   (Quest module)

  AuthService publishes UserRegisteredEvent
    -> CharacterCreatedHandler (Character module) — creates Character row

EVENT PUBLISHER (no MediatR needed yet)
  // SharedKernel/Events/
  public interface IDomainEvent { }

  public interface IEventHandler<TEvent> where TEvent : IDomainEvent
  {
      Task HandleAsync(TEvent e, CancellationToken ct = default);
  }

  public interface IEventPublisher
  {
      Task PublishAsync<TEvent>(TEvent e, CancellationToken ct = default)
          where TEvent : IDomainEvent;
  }
  // InProcessEventPublisher resolves IEventHandler<T> from IServiceProvider and calls each

  Add MediatR only when > 10 event types or pipeline behaviors (logging, validation) are needed.

DBCONTEXT STRATEGY: SINGLE AppDbContext + PER-MODULE EF CONFIGURATIONS
------------------------------------------------------------------------
Keep one AppDbContext and one migrations project (LifeLevel.Api). Do not use per-module
DbContexts — cross-module FKs (e.g. UserBossState -> UserMapProgress -> User) require a
shared context at MVP stage.

Each module owns its IEntityTypeConfiguration<T> classes.
AppDbContext.OnModelCreating calls:

    modelBuilder.ApplyConfigurationsFromAssembly(typeof(IdentityModule).Assembly);
    modelBuilder.ApplyConfigurationsFromAssembly(typeof(CharacterModule).Assembly);
    // ... one per module

Module repositories receive AppDbContext (or an IAppDbContext interface) via DI.

Future path: once past MVP, AppDbContext can be split by creating CharacterDbContext that
overrides OnModelCreating to only configure character tables. This is a mechanical refactor,
not an architectural rethink.

------------------------------------------
LifeLevel.Api          -> all modules + SharedKernel

LoginReward            -> SharedKernel

Adventure.Encounters   -> SharedKernel, Map
Adventure.Dungeons     -> SharedKernel, Map

Cycle check: none. Character depends only on SharedKernel. Identity depends only on
SharedKernel. The graph is a DAG with SharedKernel at the root and LifeLevel.Api at the top.

All cross-module port calls (ICharacterXpPort, ICharacterStatPort, ICharacterLevelReadPort,
IStreakReadPort, etc.) resolve through SharedKernel interfaces — no module except Adventure
needs a direct project reference to another module. Adventure.Encounters and .Dungeons
reference Map only for MapNode/UserMapProgress/MapEdge types used in their business logic.

KEY FILES (post-migration)
--------------------------
LifeLevel.Api/Infrastructure/Persistence/AppDbContext.cs
  All DbSets + cross-module FK relationships configured inline (cross-module FKs cannot live
  inside individual module configurations because the referenced entity types are in other
  assemblies). OnModelCreating calls ApplyConfigurationsFromAssembly for each module.

LifeLevel.Api/Application/Services/MapService.cs
  Intentionally stays in LifeLevel.Api. GetFullMapAsync loads Boss/Chest/DungeonPortal/
  Crossroads entities from multiple Adventure modules by NodeId lookup. Moving it to Map
  module would require Map → Adventure references (cycle).

LifeLevel.Api/Infrastructure/Persistence/WorldSeeder.cs
  Imperative seeder. Creates World, WorldZone, MapNode rows with relational data. Stays
  in composition root because it touches entities from multiple modules.


KNOWN MVP LIMITATIONS
----------------------
1. In-process event delivery is not guaranteed.
   Risk:   ActivityLoggedEvent is published AFTER the DB save. If StreakActivityHandler or
           QuestActivityHandler throws after the commit, the event is silently lost and
           quest/streak progress is not updated. This is a user-visible data inconsistency.
   Acceptable at MVP because: handlers are simple DB writes; failures will surface in logs
           and can be replayed manually during early development.
   Fix trigger: first production incident where a handler failure causes missing progress.
   Fix:    Transactional outbox pattern —
             a) Write the event as a row in an OutboxMessages table INSIDE the same
                EF Core SaveChanges call as the triggering entity (Activity).
             b) A background worker (IHostedService) polls OutboxMessages, delivers to
                handlers, and deletes the row on success.
             c) OutboxMessages table needs: Id, Type, Payload (JSON), CreatedAt, ProcessedAt.
           Do NOT add MediatR outbox or a message broker until this pattern proves
           insufficient (i.e., fan-out to > 5 handlers or cross-service delivery is needed).
