MODULAR MONOLITH + PORTS & ADAPTERS — LIFE-LEVEL BACKEND
=========================================================

CONTEXT
-------
The current LifeLevel.Api project is a single-project ASP.NET Core app with 28 entities,
16 controllers, 13 services, and a 41-DbSet AppDbContext. As Phase 3-9 features land
(Guild Raids, Items, Achievements, Seasonal Events, Social), the project will become
difficult to navigate, test, and reason about.

Refactoring to a Modular Monolith gives each business domain its own project with explicit
contracts, while keeping a single deployable binary and a single database. Each module uses
Ports & Adapters internally — pure domain, interface-driven application layer, EF
implementations as adapters.

Single deployable: all modules compile into one LifeLevel.Api.exe. The modules are class
libraries with no Program.cs or HTTP listener. If a module needs to become a microservice
later, the boundaries are already clean.

-------------------------------------------------------

  Domain/
    Entities/          module-owned entities only
    Enums/             module-specific enums
    Events/            domain events this module raises
  Application/
    Ports/
      In/              driving ports: IXxxService interfaces (called by controllers)
      Out/             driven ports: IXxxRepository, cross-module read ports

  Infrastructure/
    Persistence/
      Configurations/  IEntityTypeConfiguration<T> classes
      Repositories/    IRepository implementations
    {Name}Module.cs    AddXxxModule(IServiceCollection) extension method

CROSS-MODULE COMMUNICATION: TWO-TIER PATTERN
---------------------------------------------

TIER 1 — Direct interface calls (synchronous, same transaction)
  Used when: caller needs the result in the same HTTP response.

  All modules that award XP     -> ICharacterXpPort.AwardXpAsync()
  LoginRewardService            -> IStreakShieldPort.AddShieldAsync()
  CharacterService.GetProfile   -> IStreakReadPort, ILoginRewardReadPort, IDailyQuestReadPort

TIER 2 — In-process domain events (fire-and-await, after save)
  Used when: side effect does not affect the response and direct coupling would create a cycle.

  ActivityService publishes ActivityLoggedEvent
    -> StreakActivityHandler  (Streak module)
    -> QuestActivityHandler   (Quest module)

  AuthService publishes UserRegisteredEvent
    -> CharacterCreatedHandler (Character module) — creates Character row

EVENT PUBLISHER (no MediatR needed yet)
  // SharedKernel/Events/
  public interface IDomainEvent { }

  public interface IEventHandler<TEvent> where TEvent : IDomainEvent
  {
      Task HandleAsync(TEvent e, CancellationToken ct = default);
  }

  public interface IEventPublisher
  {
      Task PublishAsync<TEvent>(TEvent e, CancellationToken ct = default)
          where TEvent : IDomainEvent;
  }
  // InProcessEventPublisher resolves IEventHandler<T> from IServiceProvider and calls each

  Add MediatR only when > 10 event types or pipeline behaviors (logging, validation) are needed.

DBCONTEXT STRATEGY: SINGLE AppDbContext + PER-MODULE EF CONFIGURATIONS
------------------------------------------------------------------------
Keep one AppDbContext and one migrations project (LifeLevel.Api). Do not use per-module
DbContexts — cross-module FKs (e.g. UserBossState -> UserMapProgress -> User) require a
shared context at MVP stage.

Each module owns its IEntityTypeConfiguration<T> classes.
AppDbContext.OnModelCreating calls:

    modelBuilder.ApplyConfigurationsFromAssembly(typeof(IdentityModule).Assembly);
    modelBuilder.ApplyConfigurationsFromAssembly(typeof(CharacterModule).Assembly);
    // ... one per module

Module repositories receive AppDbContext (or an IAppDbContext interface) via DI.

Future path: once past MVP, AppDbContext can be split by creating CharacterDbContext that
overrides OnModelCreating to only configure character tables. This is a mechanical refactor,
not an architectural rethink.
