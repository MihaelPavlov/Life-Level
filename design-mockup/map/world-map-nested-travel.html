WORLD MAP · NESTED TRAVEL MODEL
A redesign of the travel loop. World-zone travel and local-map-node travel become nested layers with a single active edge at a time — you travel to a zone first, then explore its nodes. Distance overflow on arrival banks into IdleKm and is consumed by the next edge. A zone reaches Completed only after every node inside it is unlocked. Ticket: LL-043.
01STATE MACHINE
The player is always in exactly one of four phases. Transitions happen on set destination or arrival. Distance from an activity advances only the currently-active edge — never both layers at once.

Invariants

  • One active edge. Never both WorldProgress.CurrentEdgeId and MapProgress.CurrentEdgeId at the same time.
  • Distance routes, doesn't cascade. MapService.DebugAddDistanceAsync picks the active layer and forwards; the other layer stays still.
  • IdleKm banks overflow. On arrival, extra km from the activity are saved on UserWorldProgress.IdleKm (or UserMapProgress.IdleKm) and consumed at the top of the next add-distance call.
  • No chain arrival. One call resolves at most one edge. Overflow always banks, never auto-advances the next edge.
  • Zone completes on last node. After every node unlock, count nodes vs unlocked. If equal, flip UserZoneUnlock.IsCompleted = true and award ZoneCompletion XP exactly once.
  • Guards on set-destination. Server rejects a second world trip while already on a world edge. Server rejects a node destination if the node's zone ≠ CurrentZoneId, or if the user is mid-world-travel.

05IDLEKM BANK
When an activity's distance overshoots the current edge, the excess banks onto UserWorldProgress.IdleKm (or the local equivalent) and is consumed at the top of the next AddDistanceAsync call. Never lost. Never auto-applies to an edge the user hasn't chosen.
