🎯 MVP — PHASES 1–4
PHASE 1
Character & Activity Core
~3–4 weeks
▶ START HERE
The engine of the entire game. Without XP calculation, stat gains, and activity logging, nothing else works. Everything downstream (map movement, boss damage, quest progress) plugs into this layer.
BE — Backend
Character entity — finish fields: level, xp, currentXP, rank, title, class, stats (STR/END/AGI/FLX/STA)
Activity entity — type, duration (min), distance (km), calories, heartRate, loggedAt, accountId
XP engine — calculate XP from activity: type × duration + distance × multiplier
Stat gain engine — RUNNING→END+AGI, GYM→STR+STA, YOGA→FLX+STA, CYCLING→END+AGI
Level-up logic — exponential XP thresholds, level-up event triggers
POST /api/activities — log workout, returns XP gained + stat delta
GET /api/character/me — full character state
GET /api/activities — paginated activity history
FE — Frontend
Profile — Overview tab: avatar, stats bars (STR/END/AGI/FLX/STA), XP bar, level, rank
Log Workout screen: pick activity type, duration, distance, submit
XP gain animation: popup showing +XP and stat changes after logging
Level-up screen: full-screen celebration when level increases
Register + Login screens with JWT flow
App shell: bottom nav, routing
Design refs: profile-full.html (Overview tab)
PHASE 2
Daily Login & Quest System
~2–3 weeks
AFTER PHASE 1
Daily quests and login streaks are the #1 daily retention driver. Implement these early so players have a reason to open the app every day from day one. Story quests give longer-term direction.
BE — Backend
Quest entity — type (DAILY/STORY/SPECIAL), requirement, progress, reward, expiresAt
Daily quest generation — cron job: reset + generate 5 daily quests at midnight
Quest progress hook — every activity POST checks open quest requirements and updates progress
Quest completion — auto-complete when requirement met, distribute XP/item reward
Streak entity — currentStreak, longestStreak, lastActiveDate, shieldCount
Login reward table — 7-day cycle with escalating rewards (Day 7 = Rare)
GET /api/quests/daily · GET /api/quests/story · POST /api/login-reward/claim
FE — Frontend
Daily Quests tab: quest list, progress bars, complete-all bonus banner
Story Quests tab: chapter arc, current objective, lore text, reward preview
Daily Login screen: 7-day calendar, reward claim animation, streak counter
Streak broken screen: use Shield to restore or start fresh
Quest completion toast notification
Design refs: quests.html daily-login.html
⚠️ Requires Phase 1: activity logging must exist for quest progress hooks
PHASE 3
Adventure Map & Movement
~3–4 weeks
AFTER PHASE 2
The map is the visual heart of the game — it's what makes this feel like an RPG and not just a fitness tracker. Once activity → movement conversion works, the whole game world opens up.
BE — Backend
Zone entity — name, requiredLevel, coordinates, type, description, discovered
MapState entity — currentZoneId, positionKm, totalKmTraveled, lastActivityDate
Movement engine — RUNNING×1.0, CYCLING×0.7, GYM×0.2, HIKING×0.8, SWIMMING×0.5 km/km
Zone discovery — trigger when cumulative km reaches zone entry point
Seed map data — 6 initial zones with km thresholds and level requirements
GET /api/map/state · GET /api/map/zones · GET /api/map/zones/:id
FE — Frontend
Adventure Map screen: SVG zone map, player position dot, traveled path, fog of war
Zone info panel: tap zone → slide-up panel with zone details, enemies, rewards
Movement card: "Real World → Game World" conversion table at bottom of map
Zone discovery animation: fog lifts, new zone revealed
Weekly progress + streak mini-cards on map
Design refs: adventure-map.html adventure-map-detailed.html
⚠️ Requires Phase 1: movement is calculated from logged activities
PHASE 4
Boss Raid System
~2–3 weeks
AFTER PHASE 3
The MVP boss system is what makes the map feel alive and gives players a multi-day goal to work toward. Implement the core presence + damage loop first, then add the retreat mechanic.
BE — Backend
Boss entity — name, maxHp, currentHp, zoneId, state (LOCKED/AVAILABLE/ACTIVE/DEFEATED), defeatRewards
Boss state machine — LOCKED→AVAILABLE (travel to zone), AVAILABLE→ACTIVE (begin fight), ACTIVE→DEFEATED
Damage engine — RUNNING×100/km, GYM×300/session, CYCLING×60/km, HIKING×80/km
Presence check — damage only applies if character is IN boss zone (mapState.currentZoneId == boss.zoneId)
Retreat mechanic — cron: if no activity for 2 days while boss is ACTIVE, move character back one zone
GET /api/boss/active · POST /api/boss/:id/damage (called after activity log)
FE — Frontend
Traveling step: map shows boss node ahead, distance remaining, movement conversion table
Boss arrived step: full-screen boss reveal, HP bar, damage rules, "Begin Raid" CTA
Battle in progress: day counter, HP bar, damage log per day, "Log Workout" button
Victory screen: stats, rewards, new zone unlock animation
Retreat warning card on map when inactive 2 days
Design refs: boss-fight-flow.html
⚠️ Requires Phase 3: boss zones must exist on the map, character position must be tracked
🚀 FULL PRODUCT — PHASES 5–8
PHASE 5
Items, Inventory & Equipment
~3 weeks
AFTER MVP
Items make boss defeats and quest completions feel tangible. The equipment system adds a persistent progression layer — players optimise gear loadouts for different activities.
BE — Backend
Item entity — name, type (WEAPON/ARMOR/ACCESSORY/CONSUMABLE), rarity, bonuses (JSON), icon
Inventory entity — accountId, items[], maxSlots (50), equippedSlots (6)
Equipment effect engine — apply XP%, stat multiplier, boss damage bonuses when item equipped
Item reward distribution — called by boss/quest/event completion, adds item to inventory
POST /api/inventory/equip/:itemId · GET /api/inventory · DELETE /api/inventory/:itemId
FE — Frontend
Profile — Inventory tab: 4-col grid, rarity borders, filter chips, item detail panel
Profile — Equipment tab: 6-slot layout around character, selected item detail, total bonuses
Equip/unequip interaction, item rarity color system
Item drop animation on boss/quest reward
Design refs: profile-full.html (Equipment + Inventory tabs)
PHASE 6
Achievements, Titles & Ranks
~2–3 weeks
AFTER PHASE 5
Achievements provide long-term meta-progression and social bragging rights. Titles are visible on leaderboards, creating identity and status. Ranks provide structural progression beyond level.
BE — Backend
Achievement entity — name, description, tier (COMMON/UNCOMMON/RARE/EPIC/LEGENDARY), trigger, progress, unlockedAt
Achievement trigger system — event bus: activity logged, boss defeated, streak milestone, level up → check achievement conditions
Title entity — earned titles list, equipped title, unlock condition
Rank progression — Novice → Warrior → Veteran → Champion → Legend based on boss defeats + total XP
GET /api/achievements · POST /api/titles/:id/equip · GET /api/rank
FE — Frontend
Achievements screen: tier summary, category filter, in-progress bars, locked section
Titles & Ranks screen: equipped title, rank ladder, earned titles list, equip button
Achievement unlock push notification + in-app toast
Title visible on profile header and leaderboard rows
Design refs: achievements.html titles-ranks.html
PHASE 7
Social, Guilds & Leaderboards
~3–4 weeks
AFTER PHASE 6
Social features are the long-term retention multiplier. Guild raids create group accountability — if your guildmates are working out, you feel pressure to too. Leaderboards create friendly competition.
BE — Backend
Guild entity — name, members[], totalDamage, guildBossId
GuildBoss entity — shared HP pool, member damage contributions, regen mechanic (skip 2 days = +500HP)
Leaderboard service — weekly XP ranking, cached with Redis/scheduled refresh
Friends system — send/accept invite, view friend activity feed
GET /api/leaderboard · GET /api/guild/me · POST /api/guild/damage
FE — Frontend
Guild Raid screen: guild boss HP, member damage bars, SLACKING badge, shared rewards
Leaderboard screen: global + friends tabs, rank change arrows, your position highlighted
Friends search + invite flow
Mini bosses screen: list with HP bars and time remaining
Design refs: guild-raids.html mini-bosses.html
PHASE 8
Advanced Map, Events & Mounts
~4–5 weeks
POLISH PHASE
This phase adds the depth that keeps long-term players engaged. Seasonal events drive spikes of activity. Map interactions (crossroads, dungeons, NPCs) make each journey feel different. Mounts are cosmetic prestige.
BE — Backend
Crossroads system — PathChoice entity, lock logic, expiry timer, path-specific reward tables
Dungeon Portal — DungeonRun entity, floor state machine, multi-activity-type tracking, 7-day timer
NPC system — spawn after workout, 24–48h window, NPC type (MERCHANT/QUEST_GIVER/STRANGER), workout gate
Random events — event bus triggers: chest spawn, XP storm (2h window), wandering merchant
Seasonal event — SeasonalEvent entity, stage reward track, countdown, event leaderboard
Mounts — Mount entity, zone speed bonuses, cosmetic only (no XP effect)
FE — Frontend
Crossroads flow: fork node on map, choice modal, path lock UI, Valley/Ruined completion screens
Dungeon Portal: portal node, floor list, per-floor progress, boss floor gauntlet
NPC dialogs: portrait card, 3-tab interaction (Shop / Quest / Lore), XP-based shop
Random event overlays: chest popup, XP storm banner, merchant notification
Seasonal Events screen: banner, countdown, reward track, event leaderboard
Profile — Mounts tab: active mount card, stable grid, unlock requirements
Design refs: map-interactions.html crossroads-flow.html random-events.html seasonal-events.html profile-full.html (Mounts tab)
